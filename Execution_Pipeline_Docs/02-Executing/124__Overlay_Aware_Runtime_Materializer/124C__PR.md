## Verdict

**Green-light**, with five binding closeout conditions. 124C is the first
runtime-touching slice and it is correctly framed as **boring and byte-identical**:
reroute Roma's base package generation through `@clickeen/ck-runtime-materializer`
so that `same base source -> same index.html/styles.css/runtime.js -> same Tokyo
stored-bytes contract`. It honors the no-reinterpretation tenet (explicit
forbidden-additions list: no request-time composition, no URL redesign, no
storage migration, no compatibility readers, no repair, no probes) and it folds
in the prior gates' findings: Slice 5 conservatively keeps 124B fingerprint
evidence local (does not hoist — the Senior-PM 124B point), and Slice 7 mandates
a golden fixture captured *before* reroute and forbids regenerating a fixture
from the new path.

**Blast-radius shift (the reason this review is stricter than 124A/124B):** 124A
was docs-only and 124B was an unwired package — both zero runtime blast radius.
124C changes the live create/save/duplicate code path for every account. A
byte-drift bug is user-visible on the *next save* of every widget. Existing
published instances are unaffected (not re-saved), but every new save flips to
the new builder immediately — there is **no canary** for a build-path change, so
byte-parity proof is the only safety net.

## Slice-by-slice review

### Slice 1 — Call-path inventory
Sound. Naming every function moved/kept/untouched before code change is the right
discipline. No closeout condition.

### Slice 2 — Dependency wiring
Sound. Roma→materializer edge only; no reverse edge; compile + media stay in Roma.
No closeout condition.

### Slice 3 — Input adapter
Sound. **Closeout condition (deterministic input):** the adapter must build the
materializer input with **canonical/stable key order**. The current builder embeds
state via `JSON.stringify({ ... state ... })` into `runtime.js`; object key order
is insertion-order-dependent. If the adapter reconstructs the state object in a
different order than today, `runtime.js` bytes change even though meaning doesn't.
Build the input deterministically (or have the materializer canonicalize) so
output is byte-stable across runs.

### Slice 4 — Base resolver reroute (the crux slice)
- **Closeout condition (byte-exact, not semantic):** "preserve externally
  observable bytes" must mean **byte-identical** `index.html`, `styles.css`, and
  `runtime.js` — especially `runtime.js`, which embeds the `locales`/`state`
  payload and the `window.CK_WIDGETS[instanceId].state` assignment. Semantic
  equivalence is not enough: a byte change changes the SHA-256 package fingerprint
  of newly-saved packages and changes what the widget runtime executes on next
  save. That is a user-visible behavior change, which 124C explicitly forbids.
- The "preserve social-share inclusion semantics" and "package-root invalid
  failure" items are good; add explicit byte-parity assertions for them too.

### Slice 5 — Tokyo contract preservation
Sound and correctly conservative: do not change field names, object keys, content
types, or the fingerprint algorithm; keep 124B evidence local if it differs from
Tokyo's fingerprint. This is the right call (the 124B-gate deferred
fingerprint-equality to 124C, and 124C chooses "leave Tokyo unchanged"). No
closeout condition.

### Slice 6 — Error and failure mapping
Sound V1/V3/V6 discipline (no submit-on-failure, no partial success). **Closeout
condition (exhaustive error mapping):** the materializer returns its own error
types (from 124B's error contract); Roma maps them to `InstancePackageFailure`.
The mapping must be **total** — every materializer error resolves to a known
reason key. No materializer-internal error string must leak to the API response
(V7 masquerade / information leak).

### Slice 7 — Focused tests
Strong. The golden-fixture rule ("captured before reroute; do not regenerate from
the new path and call it parity") is exactly right. **Closeout condition
(dual-build comparison):** prefer a **dual-build** parity check — run the old Roma
builder and the new materializer path on identical input during reroute and assert
byte-equality — over a static captured fixture. Dual-build is stronger: a static
fixture can go stale if the input set evolves, and stale fixtures silently pass.
Cover all three files and the social-share + failure cases.

### Slice 8 — Runtime verification
Sound. **Closeout condition (deploy gated on parity):** no deploy claim until the
dual-build/fixture parity tests are green. Since every save flips to the new
builder on deploy (no canary), parity proof is the sole net. If not deployed,
closeout must state no deploy claim (already required).

### Slice 9 — Closeout gate
Correct. The "no locale artifact, no locale URL, Tokyo fingerprint behavior
unchanged, evidence not persisted as product-status" checks are the right fence
against scope creep into 124D/124E territory.

## Vectors and blast radius

| Vector | Failure mode | Blast radius |
| --- | --- | --- |
| Byte-drift in `runtime.js` (key order/formatting) | next-save widget executes different bytes → user-visible change; new fingerprint | every save, all accounts |
| Non-deterministic adapter input | package bytes vary run-to-run → fingerprint instability | per-save, intermittent |
| Semantic-not-byte "parity" test | drift accepted as "equivalent" → silent behavior change | every save |
| Stale static fixture | parity test passes against outdated expected output | every save |
| Materializer error leaks to API | internal error exposed; V7 | per-failed-save |
| Partial submit on dual failure | source without package, or package without source | per-save corruption |
