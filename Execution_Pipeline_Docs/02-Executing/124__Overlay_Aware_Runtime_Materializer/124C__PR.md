## Slice-by-slice review (revised)

Slice 1 (inventory) and Slice 2 (wiring) are sound. Slice 2 correctly keeps the
Bob-route call and Tokyo media resolution in Roma and forbids reverse edges.

### Slice 3 + Slice 4 — Input adapter + base reroute (THE crux)
- **Closeout condition 1 (byte-exact, targeting the real serialization path):**
  the drift risk is **not** the adapter wrapper — it is the **state object's key
  order**. Verified chain: `materializeConfigMedia` (`ck-contracts/src/index.ts`)
  reconstructs every node via `Object.entries` + fresh object literals + spreads,
  producing an insertion-order-determined key order that becomes `state`;
  `account-instance-public-package.ts:295-302` then embeds it via
  `JSON.stringify({ instanceId, baseLocale, locales })` into `runtime.js`, and
  Tokyo's SHA-256 fingerprint (`package-files.ts:93-104`) is over those raw
  bytes. A single state-key reorder flips the fingerprint on every post-reroute
  save and changes what the widget executes. **The byte-safe path is one of:**
  (a) pass the *already-stringified* state payload through untouched, or
  (b) have the materializer canonicalize AND prove the canonical form equals
  today's `JSON.stringify` output. "Build the adapter input deterministically"
  (Rev 1's wording) targets the wrong layer — the wrapper, not the state graph.
- **Closeout condition 2 (dual-build, across all widget types):** prefer
  dual-build (run old `buildSavedWidgetPublicPackage` and the new materializer on
  identical input, assert byte-equality) over a static fixture. Dual-build is
  stronger because it covers the duplicate route's Tokyo-round-tripped configs
  and arbitrary request bodies, not one frozen input. **It must run across all
  widget types currently saved in the cloud-dev admin account** —
  `stampPackageRoot`/`resolveProductPath` are widget-shape-sensitive, so
  synthetic fixtures or a single widget type do not prove parity.

### Slice 5 — Tokyo contract preservation
Sound and correctly conservative (no field/key/content-type/fingerprint-algorithm
change; 124B evidence local if it differs). Keeping Tokyo 100% unchanged is the
right cohesion call — a storage-contract change is CDN-wide blast radius, and a
"boring reroute" must not mutate it.

### Slice 6 — Error and failure mapping
Sound V1/V3/V6 discipline. **Closeout condition 3 (total error mapping):** the
materializer returns its own error types; Roma's mapping to `InstancePackageFailure`
must be **total** — every materializer error resolves to a known reason key; no
internal error string leaks to the API (V7).

### Slice 7 — Focused tests
Strong golden-fixture rule. **Closeout condition 4 (harness):** see execution
prerequisite 2 — the tests need a runner that does not yet exist. **Closeout
condition 5 (breadth):** parity across all widget types in cloud-dev, not
synthetic (see condition 2).

### Slice 8 — Runtime verification
**Closeout condition 6 (deploy gated on parity):** no deploy claim until
dual-build/fixture parity is green across all widget types. There is **no canary**
for a build-path change — every save flips immediately on deploy — so parity
proof is the sole net. If not deployed, closeout states no deploy claim.

### Slice 9 — Closeout gate
Correct fence (no locale artifact/URL, Tokyo fingerprint unchanged, evidence not
persisted as product-status).

## Vectors and blast radius

| Vector | Failure mode | Blast radius |

| Parity tested on one widget type only | widget-shape-specific drift (`stampPackageRoot`/`resolveProductPath`) slips through | per-widget-type, silent |
