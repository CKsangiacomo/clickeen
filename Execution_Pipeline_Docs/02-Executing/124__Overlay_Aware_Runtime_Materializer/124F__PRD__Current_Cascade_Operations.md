# PRD 124F - Current Cascade Operations

Status: EXECUTING
Parent: `124__MAMA__Overlay_Aware_Runtime_Materializer_Program.md`
Depends on: 124A GREEN, 124B/124C/124D/124E CODE-DELIVERED for every runtime behavior 124F calls
Owner: Roma account commands

## Purpose

Wire the current executable cascade operations for account widget instances:

- source instance save;
- locale overlay write through current translation generation;
- active locale add/remove;
- account policy value that affects current artifact output.

124F does not create a broad dependency engine. It wires direct account commands
that already own the mutation.

## Cascade Law

Clickeen cascade is command-owned and coordinate-exact:

```text
source command changes truth -> same command resolves affected artifact coordinates
```

The system must not claim an affected coordinate is current when that coordinate
was not regenerated, deleted, or explicitly named as incomplete.

## Non-Reinterpretation Tenet

124F must not reinterpret current cascade into a global dependency system.

Forbidden additions in 124F:

- dependency graph services;
- event buses;
- schedulers;
- background cascade jobs;
- watchers;
- storage scans that discover work;
- lifecycle/status files;
- readiness markers;
- repair/backfill routines;
- best-effort hidden retries;
- broad schema/widget/Dieter/materializer fan-out;
- page/site/email cascade.

## Precondition Gate

124F may begin only after these contracts exist in code, not only as
plan-green PRDs:

1. 124D names the locale package materialization helper.
2. 124D names the locale package delete helper.
3. 124D names the materialization/delete response shape.
4. 124E names whether locale entry refresh uses command-triggered purge or
   short TTL only.
5. 124D/124E helper behavior is code-delivered, not only plan-green, wherever
   124F calls those helpers.

If any precondition is missing, 124F stops before implementation. The agent
must not invent helper names, response fields, purge behavior, or refresh
semantics.

## Execution Dependency Law

Plan-green is not enough for 124F call sites. 124F depends on 124B/124C for the
base materializer/reroute behavior and calls the materialization, delete,
serving, cache, and purge contracts produced by 124D/124E, so those contracts
must exist in code before Roma source-save, translation, or locale settings
routes depend on them.

If 124D/124E have not delivered code for a needed helper or serving behavior,
124F records the missing dependency and stops at that slice. It must not
create a replacement helper, shadow storage coordinate, queue, watcher, status
file, or cache rule to make the plan appear executable.

### Goal

### Steps

   - create instance;

   - duplicate instance.
2. Inventory current overlay commands:
   - `POST /api/account/instances/{instanceId}/translations/generate`;
   - account active-locale settings follow-up for added locales.
3. Inventory current active-locale removal command:
   - `PUT /api/account/locales`.
4. Inventory current account policy inputs that affect generated bytes.

6. For each command, record:
   - source truth changed;
   - affected artifact coordinates;

### Compliance Rationale

## Slice 2 - Source Save Cascade

### Goal

When saved source changes, base and active locale artifacts are updated or exact
incomplete coordinates are reported.

### Steps

1. Keep the existing source save command order through base package creation:
   - load account locale state;
   - compile widget package;
   - run account instance policy gate;
   - materialize base public package through 124C;
   - materialize source artifacts;
   - save source/base package through Tokyo.
2. If base source/package save fails, do not start locale cascade.
3. Run locale cascade only when saved source or byte-affecting inputs actually
   changed, or when the command explicitly requests regeneration.
4. Do not regenerate overlays/packages on no-op saves unless current product
   behavior already treats that save as a regeneration command.
5. After source/base save succeeds, resolve active non-base locales from the
   account locale state already loaded by Roma.
6. If there are no active non-base locales, return source save success with an
   explicit empty locale-cascade result.
7. If active non-base locales exist, regenerate overlays for those active
   locales through the existing Roma Translation Agent operation.
8. After each overlay generation succeeds, materialize generated locale package
   artifacts through the 124D helper.
9. Process active locales in deterministic active-locale order.
10. The save response must explicitly expose whether translation generation was
    invoked, completed, skipped, or failed per locale.
11. No background or hidden retry continues after the response.
12. On first locale cascade failure, stop remaining locale processing and return
   primary source/base save success with an explicit locale-cascade failure
   payload naming:
   - completed locale coordinates;
   - skipped locale coordinates;
   - failed locale coordinate;
   - failed phase: translation generation, overlay read, materializer, package
     write, or purge/entry refresh if 124E selected purge.
13. Affected routes must not return a top-level success shape that existing
    callers/UI could interpret as all requested source and locale work complete
    when locale cascade failed.
14. Do not claim all active locale artifacts are current unless every active
   non-base locale completed overlay generation and package materialization.
15. Do not apply old overlay values to changed source and call that current
    unless 124A/124D explicitly defines that exact operation as current.
16. If source/base save succeeds and locale cascade fails, old locale packages
    must not be claimed current. Public locale serving must either fail closed
    through 124E evidence/source-pointer mismatch or continue only where 124E
    evidence still matches the published source pointer. Roma must not repair
    this on visitor requests.
17. Bound the synchronous source-save cascade before implementation:
    - active non-base locale count is governed by `l10n.locales.max`
      (`free: 0`, `tier1: 3`, `tier2-4: 28`);
    - source save is one instance times active non-base locales;
    - the worst account-tier locale surface for this command is one instance
      times the tier2-4 cap of 28 active locales;
    - record the expected worst-case work against the Cloudflare Pages request
      budget for the route being changed.
18. If that bounded work cannot fit inside the current request budget, stop
    before implementing a hidden partial sync path. A later PRD may define an
    explicit account command with its own authority, response shape, cost
    ceiling, and deploy evidence; 124G/later is the named escape valve for
    broader async/account-wide cascade work. 124F must not add a background
    queue or watcher to hide the cost.

### Output

Existing source save updates base artifacts and either updates active locale
artifacts or names exact incomplete locale coordinates.

### Compliance Rationale

This is compliant because source truth drives dependent generated artifacts in
the same account command. It avoids stale locale packages being silently treated
as current.

## Slice 3 - Instance Create And Duplicate Cascade

### Goal

Keep create and duplicate behavior explicit without inventing locale work.

### Steps

1. Instance create keeps current behavior:
   - create source;
   - create base package;
   - no locale overlays exist yet for the new instance.
2. Return an explicit empty locale-cascade result when the create response shape
   needs to expose cascade state.
3. Instance duplicate keeps current behavior:
   - duplicate is a new account operation;
   - duplicate regenerates base package for the new instance id;
   - duplicate does not copy locale authority from the source instance.
4. Do not copy overlays from the source instance.
5. Do not copy generated locale packages from the source instance.
6. If product later requires duplicate-with-locales, that is a separate account
   command and not hidden in 124F.

### Output

Create/duplicate remain clean base-source commands with no accidental locale
carryover.

### Compliance Rationale

This is compliant because new instance identity must not inherit translated
truth by copying artifacts from another coordinate.

## Slice 4 - Overlay Write Cascade

### Goal

When a locale overlay changes through current generation, only that locale
artifact is regenerated.

### Steps

1. Keep Translation Agent responsible only for overlay value production.
2. Keep Tokyo responsible for exact overlay file writes.
3. After Roma confirms overlay generation succeeded for a locale, call the 124D
   locale materialization helper for that same locale.
4. Do not regenerate unrelated locales.
5. Do not regenerate base package from an overlay write.
6. If materialization fails after overlay write, keep the overlay file and
   return explicit coordinate failure.
7. Do not claim generation success unless overlay write and locale package
   materialization both completed for every requested locale.

### Output

Overlay generation updates only affected locale package coordinates.

### Compliance Rationale

This is compliant because overlay writes are locale-scoped. The cascade follows
the changed coordinate and avoids broad fan-out.

## Slice 5 - Active Locale Add/Remove Cascade

### Goal

Preserve account active-locale semantics while updating affected artifacts.

### Steps

1. Keep active-locale setting writes owned by Roma account settings.
2. The affected instance coordinate list for active-locale add/remove must come
   from an existing Roma-owned account route/helper explicitly named in Slice 1.
3. 124F must stop if the command cannot name exact affected instance
   coordinates without storage scans or inferred discovery.
4. When active locales expand:
   - save the account locale setting as current account truth;
   - generate overlays for added active non-base locales through Translation
     Agent;
   - materialize generated locale package artifacts through 124D;
   - report explicit follow-up failure if any affected coordinate fails.
5. When active locales shrink:
   - save the account locale setting as current account truth;
   - delete exact overlay files for removed locales through Tokyo;
   - delete generated locale package artifacts for removed locales through the
     124D storage/delete path;
   - purge or short-TTL-refresh entry URLs according to the 124E choice;
   - report explicit follow-up failure if overlay deletion, package deletion,
     or purge/entry refresh fails.
6. Do not claim removed locales stop serving until generated package deletion
   succeeds.
7. Do not add visitor-time active-locale checks to public serving.
8. Bound the synchronous active-locale settings follow-up before implementation:
   - current code already loops through every saved account instance returned by
     Tokyo and every added/removed non-base locale in
     `reconcileAccountLocaleOverlays`;
   - the coordinate count is `instance count * changed locale count`;
   - `changed locale count` is capped by `l10n.locales.max`, but instance count
     must come from the existing Roma/Tokyo account instance list and must not
     be guessed.
   - the worst account-tier locale surface is tier2-4 at the 28-locale cap
     across every saved account instance returned by Tokyo.
9. If the current product policy does not bound instance count tightly enough
   for this synchronous route, record that as an execution stop for sync
   expansion. Do not silently rely on Cloudflare timeout behavior, partial
   completion, or a hidden async follow-up. 124G/later is the named escape
   valve if this work cannot fit the direct request budget.

### Output

Active-locale settings produce exact overlay/package add or remove effects for
current widget instances.

### Compliance Rationale

This is compliant because account settings remain the account truth while
artifact follow-up is explicit and coordinate-scoped. Public serving remains
stored-byte only.

## Slice 6 - Account Policy Byte-Affecting Changes

### Goal

Handle only current account policy changes that actually affect generated
artifact bytes or allowed active locales.

### Steps

1. Inventory current policy values used by package generation.
2. 124F may implement policy cascade only for:
   - a named existing policy command;
   - a named byte-affecting policy value;
   - an exact affected coordinate list.
3. If no such current command/value/list exists, record policy cascade as
   no-op/out of scope and do not add policy cascade code.
4. 124F does not add a new policy-change command, watcher, or account-wide
   policy fan-out.
5. If a policy value only gates future actions and does not change current
   artifact bytes or active locale allowance, do not cascade artifacts.
6. If a policy change removes allowed active locales, route through the same
   active-locale shrink behavior from Slice 5.
7. If a policy change changes generated package bytes for an existing instance,
   route through the same source-save/base-plus-active-locale cascade behavior
   for the affected instance coordinates.
8. Do not create a general policy-cascade engine for possible future products.
9. Do not infer upgrade/downgrade artifact work without a named policy value,
   owning command, and affected coordinate list.

### Output

Current policy-affecting changes use the same direct command cascade as source
or active-locale changes.

### Compliance Rationale

This is compliant because policy cascade is limited to declared byte-affecting
truth. It avoids speculative product machinery.

## Slice 7 - Response Contract

### Goal

Make command responses honest without inventing status files.

Current source save returns a bare `{ ok: true }`. Any 124F change that adds
locale cascade behavior to source save is therefore a caller/UI contract
change, not merely an internal detail.

The changed response must be visible to the operator through the named
Roma/Bob caller. If source save succeeds but locale cascade partially fails,
the operator must see which locale/package coordinate failed and what remains
incomplete. Returning honest JSON that no product surface handles is not enough
for 124F closeout unless the UI handoff is explicitly assigned to a named
follow-up owner.

### Required Shape

For commands that save the primary truth and then run artifact follow-up, the
response must distinguish:

- primary truth saved;
- affected coordinates completed;
- affected coordinates skipped;
- failed coordinate;
- failed phase;
- cache refresh/purge completion when 124E selected purge;
- full cascade success as true/false, or the existing equivalent `ok` field.

### Steps

1. Reuse existing response envelopes where possible.
2. Add cascade detail only to routes whose behavior changes.
3. Do not persist cascade status as a product artifact.
4. Do not return a plain success response when required affected coordinates
   failed.
5. Do not call incomplete locale cascade “complete,” “current,” or “published.”
6. Command responses must distinguish `artifact updated/deleted but cache
   refresh failed` from full cascade success when purge is part of the cascade.
7. Name the existing Roma/Bob consumer that renders or handles the cascade
   detail before changing a response shape. If no current UI renders it, 124F
   may still return the detail to preserve truth, but closeout must state the
   operator-visible UI handoff explicitly and must not call the product behavior
   complete until Roma/Bob makes changed cascade outcomes visible to the
   operator or a named follow-up PRD owns that UI.
8. Do not create a ledger, status file, operation table, polling surface, or
   lifecycle object to make cascade details visible.

### Output

Command responses expose exact coordinate completion without sidecar state.

### Compliance Rationale

This is compliant with V1-V8, especially silent omission and partial-success
masquerade.

## Slice 8 - Focused Tests

### Goal

Prove command-owned cascade behavior.

### Required Tests

1. Source save:
   - base save succeeds and no active locales returns empty cascade;
   - active locales trigger translation generation and locale materialization;
   - first locale failure returns completed/skipped/failed coordinates;
   - old overlay values are not treated as current for changed source unless
     the closed 124A/124D contract allows it.
   - no-op saves do not regenerate overlays/packages unless the command
     explicitly requests regeneration.
   - source save does not expose a top-level success shape that existing callers
     can read as full source-and-locale success when locale cascade failed.
   - source-save cascade cost is bounded by one instance times active non-base
     locales and the route stops before hidden partial sync if that work cannot
     fit the request budget.
2. Overlay generation:
   - successful overlay write triggers same-locale package materialization;
   - materialization failure after overlay write returns explicit coordinate
     failure and no full-success claim.
3. Active locale add:
   - added locale generates overlay and package artifact;
   - added locale failure reports follow-up failure.
4. Active locale remove:
   - removed locale deletes overlay and generated package artifact;
   - package deletion failure reports follow-up failure;
   - response does not imply removed locale stopped serving unless deletion
     succeeded.
   - artifact delete succeeds while purge fails, and the response distinguishes
     cache refresh failure from full cascade success when 124E selected purge.
   - active-locale settings follow-up records the `instance count * changed
     locale count` cost surface and does not claim full success after timeout or
     unattempted coordinates.
5. Policy changes:
   - no cascade for non-byte-affecting policy values;
   - no policy cascade code when no named existing command/value/coordinate list
     exists;
   - active-locale entitlement shrink uses active-locale shrink behavior when
     implemented.
6. Dependency guard:
   - no new broad dependency service, event bus, scheduler, watcher, or status
     file code.

### Output

Focused Roma tests, with Translation Agent/Tokyo tests only for touched
surface-specific behavior.

### Compliance Rationale

This is compliant because tests prove direct command behavior. They do not
become runtime probes or hidden validators.
