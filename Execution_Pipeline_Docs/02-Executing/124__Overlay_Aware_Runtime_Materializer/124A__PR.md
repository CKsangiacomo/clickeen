## Verdict

**124A is a sound, compliant, executable contract-lock plan.** It locks the
seed contract (schema-token identity, evidence, URL/CDN, failure, Translation
Agent compatibility, preview/public parity) before any runtime code, over
*existing* artifacts, with the no-reinterpretation tenet stated verbatim and
honored throughout. It is the right shape for the highest-risk slice of the
program. Green-light recommended, subject to the watch-items below — none of
which change the plan, all of which are decisions 124A itself must close
definitively (not leave as a menu).

The tenet check passes: 124A does **not** reinterpret PRD 124 into an ideal
system and add machinery. It forbids a Schema service, token registry, identity
database, product-status store, readiness ledger, and compatibility reader, and
it reconciles token identity against the existing `SavedTextField.identityKey` /
`fieldPattern` substrate. That is the correct discipline for a seed slice.

## Why 124A is load-bearing (blast radius)

124A produces no code — only locked contracts. But 124B–124H implement against
those contracts, and (per the MAMA atomic-cascade-risk law) a base contract
mistake propagates `schema → token → overlay → artifact → surface/app`. So a
124A lock error is not a 124A bug; it is a program-wide, forward-propagating
defect. That is exactly why this slice is contract-first, and why every
unresolved item must be either closed or assigned to a named later sub-PRD. The
plan understands this.

## Slice-by-slice review

### Slice 1 — Authority & artifact inventory
- **Locks:** the current files/services/docs owning schema, source, overlay, package, preview, serving.
- **Executable:** yes — pure read-and-cite; output is a table.
- **Compliance:** Tenet 2 (named authorities), MAMA execution doctrine ("operate only declared truth"). Starting from current truth is the antidote to idealizing a new schema system.
- **Watch-item:** none material. This slice is correctly boring.

### Slice 2 — Schema-token identity contract
- **Locks:** scalar identity = path; repeated identity = path + `arrayItemIdentity` + saved ids; one canonical representation for 124B; failure semantics for missing/duplicate id, unmappable overlay key.
- **Executable:** yes — and de-risked, because `identityKeyForField()` + `RepeatContext` + `SavedTextField.identityKey/fieldPattern` already exist. Slice 2 is reconciliation, not greenfield.
- **Compliance:** anti-overarchitecture tenet (identity scoped to repeaters; scalar stays path-keyed), source-truth fidelity.
- **Watch-items (decide, don't menu):**
  1. **Reuse vs supersede `identityKey`.** An `identityKeyForField()` implementation already exists. 124A must pick ONE: canonical identity = the existing `identityKey` (reuse, zero migration) or a new bracket-id form like `faq.sections[pricing].…` (supersede, migration + dual-read risk). Reuse is strongly preferred unless the existing form is structurally wrong — superseding introduces exactly the compatibility-reader machinery 124A forbids.
  2. **Saved-id presence assumption.** The repeated rule assumes saved instances carry stable ids on every repeater item. If any saved instance lacks them, locale materialization fails closed for those items. 124A must state whether that fail-closed is acceptable as the data-readiness contract, or whether id-backfill is a prerequisite (owned elsewhere).

### Slice 3 — Translation Agent overlay-key compatibility (THE crux)
- **Locks:** one decision — `compatible | requires_full_overlay_key_chain_change | scalar_only_initially`.
- **Executable:** yes, but it is the highest-leverage decision in 124A because it determines whether locale materialization can touch repeated fields at all.
- **Compliance:** fail-visible / no silent substitution (V1); refuses to claim capability the current overlay shape can't support.
- **Watch-item (verified context):** the system has a real contradiction to resolve — `identityKeyForField()`/`RepeatContext` exist (suggesting identity addressing is built), but the Babel doc example shows positional `sections.0.faqs.0.question`, and Tokyo overlay storage is path-keyed. 124A Slice 3 step 1 ("inspect current Translation Agent output shape") is what resolves it. The decision must be evidence-backed from actual runtime output, not doc prose. If the truth is `scalar_only_initially`, that is a legitimate, explicitly-claimed product limitation — NOT a gap to hide — and 124D's locale scope follows from it.

### Slice 4 — Artifact evidence contract
- **Locks:** evidence fields = real independently-moving inputs only; fingerprint method; storage location; exclusions.
- **Executable:** yes.
- **Compliance:** MAMA evidence law ("do not invent separate fingerprints for concepts that do not move separately"); preserves Tokyo as stored-byte authority and Babel bodies as pure value maps.
- **Watch-item (decide, don't menu):** **single-source fingerprint computation.** Evidence is checked at serve time (Tokyo) against fingerprints computed at build time (Roma/materializer). If two implementations compute the JSON canonicalization differently, evidence fails spuriously (false cascade) or passes wrongly (stale serve). 124A must lock that fingerprint computation lives in **one shared module** (the materializer package or `ck-contracts`), used by both build and serve validation — plus the exact canonicalization rule (sorted keys / RFC 8785), hash algo, and byte-order. Two reimplementations is a V1/V6 vector.

### Slice 5 — Public locale URL, artifact paths, CDN
- **Locks:** pretty-URL entry (short TTL) + fingerprinted immutable support files; cache headers; entry refresh mechanism; materializer must not purge.
- **Executable:** yes.
- **Compliance:** MAMA CDN law; preserves serving economics (no visitor-time materialization/overlay reads).
- **Watch-item (decide, don't menu):** **does the purge operation path exist today?** Step 8 offers a menu (existing purge path OR short-TTL-only until a later PRD). The MAMA doc asserts "Roma owns entry artifact update and purge" — so the *owner* is named, but 124A must confirm the *operation* exists. If it doesn't, lock short-TTL as the explicit interim and name the managed-Cloudflare-operation gap for a later PRD. Do not leave it as an open menu — a stale-locale-serving window (V1-ish) depends on which branch is true.

### Slice 6 — Failure contract
- **Locks:** exact failure for ~14 conditions; fallback always `no` for requested locale; repeated fields explicitly non-claimed (not silently omitted) if deferred.
- **Executable:** yes.
- **Compliance:** V1 (no silent substitution), V3 (no silent omission), Tenet 3 (no fallback).
- **Watch-item:** none — this is the strongest slice. The "explicitly non-claimed, not silently omitted" rule for deferred repeated fields is exactly the V3 guard.

### Slice 7 — Preview/public parity
- **Locks:** shared resolver code preferred; conformance-only allowed with recorded justification + a conformance suite covering scalar/repeated/missing/unexpected/no-fallback cases.
- **Executable:** yes — but shared code across Bob preview (browser JS) and materializer (server TS) is genuinely hard; conformance-only is the likely outcome.
- **Compliance:** preview-is-a-product-promise; no runtime probes or QA ritual.
- **Watch-item (decide, don't menu):** if conformance-only is chosen, the conformance suite **must be a CI gate**, not a manual check — two resolver implementations will drift otherwise (silent preview≠publish divergence = V1/V6). 124A should lock that the suite runs in CI on every change to either resolver.

### Slice 8 — Closeout gate
- **Locks:** all slices complete, no runtime code changed, unresolved items assigned, V1–V8 audit run, no placeholder/compatibility language left unassigned.
- **Executable:** yes.
- **Compliance:** MAMA granularity-is-product-safety; "make invalid states unrepresentable, not detectable later by process."
- **Watch-item:** none — correct gate. The "no unresolved placeholder unless assigned to a named sub-PRD with explicit limitation" rule is the right closeout bar.
