## Slice-by-slice review (revised)

### Slice 1 — Inventory
Boring and correct. No change from Rev 1.

### Slice 2 — Schema-token identity contract
- **Locks:** scalar = path; repeated = path + `arrayItemIdentity` + saved ids; one canonical representation.
- **Binding closeout conditions (verified-grounded):**
  1. **Reuse, do not reinvent.** Canonical identity must reconcile with the existing `identityKeyForField()` / `SavedTextField.identityKey` and the existing `saved_text_field_identity_*` reason keys. Slice 2 must state whether canonical identity = the existing `identityKey` (reuse) or a new bracket-id form (supersede). Reuse is strongly preferred; superseding risks the compatibility-reader machinery 124A forbids.
  2. **`identityKey` reuse is real migration work, not zero-cost.** `identityKey` is an internal contract helper; it is **not** wired to any user-facing overlay surface today (Translation Agent emits positional `item.path`; Tokyo validates positional paths). So "reuse identityKey" means 124D must change the producer/consumer chain to emit and accept identity keys. 124A must state this explicitly so 124B/124D do not assume a drop-in reuse and then reach for a compatibility reader when it isn't.
  3. **State the live V1 hazard the contract closes.** Today, a builder who reorders a repeated field (e.g. an FAQ section) after overlays are generated silently gets the wrong translation under the wrong item in preview (and, once locale serving ships, in serve). That is a present-day V1 (silent substitution) on exactly the surface Babel protects. The identity contract exists to make that fail-closed instead of silent. Naming it keeps the contract honest about what it's for.

### Slice 3 — Translation Agent overlay-key compatibility (THE crux)
- **Binding closeout condition:** **drop `compatible` from the menu.** Verified end-to-end: Translation Agent keys overlays by concrete positional `item.path` (`index.ts` → `values.ts` → `previewL10n.js`). For repeated fields, `compatible` is not available. The decision is binary and evidence-backed:
  - `scalar_only_initially` — locale cascade ships for scalar translatable text only; repeated-field locale cascade is an **explicitly claimed limitation** until the overlay-key chain changes; or
  - `requires_full_overlay_key_chain_change` — owner + exact output/read/write shape named for Translation Agent, Roma translation routes, Tokyo overlay validation, Bob preview, and the Babel/OverlayArchitecture docs.
- Given 124D's scope, `scalar_only_initially` is the expected outcome. Either way, 124A must not leave `compatible` on the menu as a wish.

### Slice 4 — Artifact evidence contract
- **Binding closeout conditions (verified-grounded):**
  1. **Extend `publicPackageFingerprint`, do not build parallel.** A serve-time SHA-256 fingerprint gate already runs (`package-files.ts:52,103` + `clk-live-routes.ts:118-131`). Slice 4 must name this mechanism and extend it, not describe evidence as greenfield.
  2. **Split the 8-field evidence into three tiers** (not one flat list):
     - **serve-checked** — `generated package fingerprint` (already gated at serve; reuse);
     - **build-recorded** — `source fingerprint`, `overlay fingerprint` (non-base), `schema/widget contract fingerprint`, `materializer contract version`, `support-file fingerprints` — stored as R2 object/package metadata, checked where the serving rule requires;
     - **echo-only** — response-payload evidence as proof, never product truth.
  3. **Reconcile the two hashes.** SHA-256 (`package-files.ts:103`) for package/artifact fingerprints (deterministic, crypto-grade, the serve-gate uses it); FNV-1a (`translated-value-primitives.ts:458-464`) for internal identity keys (non-crypto). State explicitly which hash backs which evidence field, that they are not interchangeable, and that each is owned by **one shared module** used by both build and serve (no second implementation — divergence is a V1/V6 vector).
  4. **JSON canonicalization rule** locked (sorted keys / RFC 8785) so build-side and serve-side agree.

### Slice 5 — Public locale URL, paths, CDN
- **Binding closeout conditions (verified-grounded):**
  1. **Pick the locale URL shape in 124A — do not defer.** Verified: `parseClkLivePath` handles account/instance only; there is **no locale route** in `clk-live-routes.ts` today. So "use the current public serving authority" defers to an authority that does not exist for locales. 124A must choose path-segment vs query vs prefix explicitly (Step 0 owns the exact string).
  2. **Purge: code exists, config does not → lock the honest interim.** Verified: `purgeClkLiveEntryCache` exists and is wired into saves (`operations.ts:77-116,366,391`), but `CLOUDFLARE_ZONE_ID` / `CLOUDFLARE_API_TOKEN` are **not bound** in `tokyo-worker/wrangler.toml`, are marked optional in `tokyo-worker.md`, and there is **no `cf:purge` repo command**. The code fails closed (`503 tokyo.errors.publicCache.purgeConfigMissing`) on missing config — correct, but it means managed purge is **not proven operational** today. 124A must lock **short-TTL entry freshness as the serving interim** and name "bind purge config + add a repo purge operation" as a documented gap for 124E/124F. Do not assert purge as a current capability.
  3. Materializer must not purge (already stated); Tokyo serves stored evidenced bytes only (already stated).

### Slice 6 — Failure contract
Strongest slice; no change. `Fallback = no` for requested locale; deferred repeated fields explicitly non-claimed, not silently omitted (V3 guard). Reuse the existing `saved_text_field_identity_*` reason keys where Slice 2 failures overlap.

### Slice 7 — Preview/public parity
- **Binding closeout condition:** **the conformance suite must be a CI gate.** Verified: preview and public already run **two divergent resolver implementations** today — `previewL10n.js` (browser JS) silently returns on type mismatch, while the server path (`translated-value-primitives.ts`) throws. Shared code is preferred but unlikely across the browser/server split; conformance-only is the realistic outcome, and with two implementations it will silently drift unless the suite runs in CI on every change to either resolver. Lock CI-gating, not manual QA.

### Slice 8 — Closeout gate
Correct. Add: V1–V8 audit must specifically re-check the live reorder→wrong-translation hazard (Slice 2) is fail-closed under the locked contract.
