## Notes (non-blocking)

1. **The addendum locks AS-IS, defers TO-BE — correctly.** It locks current
   reality (concrete paths, single `publicPackageFingerprint`, short-TTL CDN for
   all three files) and defers the target state (reorder-safe token cascade,
   multi-fingerprint evidence, two-tier immutable CDN) to named later PRDs. This
   is the right discipline for a contract lock: lock what 124B executes now, do
   not lock future-state machinery that isn't built.
2. **§15 V1–V8 self-audit must be independently confirmed, not accepted.** The
   addendum self-asserts all eight clean; the three-role gate independently
   verifies the load-bearing claims (it does). Don't let the self-audit become a
   substitute for the gate.
3. **Addendum drift risk.** Once later slices (124C/124E) change runtime behavior,
   the addendum's "current" locks (e.g., CDN headers, single fingerprint) must be
   re-validated against the new code or they describe a stale system. The team
   should treat the addendum as versioned alongside the code it locks.
