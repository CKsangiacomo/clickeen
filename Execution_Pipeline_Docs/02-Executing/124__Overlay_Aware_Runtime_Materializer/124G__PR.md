## Verdict

**Green-light.** 124G is an **audit + contract slice, not an implementation** —
and that is exactly the right discipline. It inventories the broad dependencies
(widget spec/editable-fields/shared runtime/Dieter/materializer/account assets),
classifies each (sealed-into-bytes vs external-reference vs schema-identity vs
evidence), defines the sealed-bytes and external-reference rules, the
evidence/version compatibility contract, the unmarked-artifact rule, and —
crucially — **the gate any future broad re-resolution PRD must pass** (named
dependency, evidence field, coordinate source, operator authority, command path,
max-coordinate-count, cost ceiling, failure shape, deploy evidence, V1–V8 audit)
while implementing **no** mass re-resolution itself. It honors the
no-reinterpretation tenet (forbids broad dependency service, event bus,
scheduler, scanner, status store, hidden mass re-resolution). No blockers; one
residual-risk note + a dependency.

## Why "audit, not engine" is correct

This is the disciplined answer to the concern I raised at the very start of this
program ("cascade at scale has no executor — a Dieter/widget/materializer change
stale-s the whole public surface"). 124G does **not** build that executor. It
audits the dependency surface and defines the gate a future executor must pass.
That is right: prove the contract (what's sealed, what's external, what evidence
detects staleness) before adding a managed mass-operation surface. Building the
fan-out engine now — before the evidence/version contract is real — would be
exactly the overarchitecture the tenet forbids.

## Residual risk (explicit, accepted)

**After 124G, system-wide fan-out still has no executor.** A base-layer change
(widget software, Dieter token, materializer contract version) can stale every
published artifact, and nothing regenerates them en masse — each artifact is only
re-resolved when its owning account command runs (124F) or fails closed at serve
(124E evidence mismatch). 124G makes this **explicit and gated** (Slice 6: any
future broad operation needs a named authority + blast-radius controls), not
hidden. That is the honest state; the residual risk is accepted until a future
broad-re-resolution PRD builds the engine under 124G's gate.

## Slice-by-slice

- **Slice 1 (inventory):** sound. Names every broad dependency and forces
  classification (sealed / external-ref / schema-identity / evidence) or stop.
- **Slice 2 (sealed-bytes):** sound. Stored package bytes don't auto-mutate on a
  dependency deploy; explicit command regenerates or serving fails closed on
  evidence. No visitor-time source comparison.
- **Slice 3 (external references):** sound. Mutable external refs (`/dieter/**`,
  `/fonts/**`, account asset URLs) flagged as broad-dependency risk; no
  visitor-time checks; no silent copy-in.
- **Slice 4 (evidence/version):** sound. Materializer-contract-version
  compatibility rules; no package-body status fields; no public-serving evidence
  recomputation; no invented expected-version store. **Dependency:** "confirm
  generated packages record source/widget/materializer fingerprints required by
  124A" rests on the 124A §9.1 carry-forward — 124C must actually compute/persist
  source/schema fingerprints (only `publicPackageFingerprint` exists today), or
  this audit's evidence detection is unenforceable.
- **Slice 5 (unmarked artifacts):** sound V5. No silent fallback; marked/unmarked
  mix fails closed per existing Tokyo behavior; no compatibility readers; no
  rewrite.
- **Slice 6 (future authority gate):** the strongest slice — defines the
  10-field gate any future broad operation must pass, and explicitly states 124G
  implements none of it.
- **Slices 7–9 (deploy evidence, audit tests, closeout):** sound. "No
  implementation" check (no runtime routes/scanners/status files/mass-mutation)
  is the right purity fence.

## Vectors and blast radius

| Vector | Failure mode | Blast radius |
| --- | --- | --- |
| Base-layer change with no executor | whole public surface goes stale → mass 404 (fail-closed) or mass stale-serve | **system-wide, all accounts** (accepted residual until future PRD) |
| Evidence unenforceable (source/schema fp not computed) | staleness undetectable | audit-only detection fails |
| Mis-classified sealed vs external | wrong regeneration/no-regeneration decision | per-dependency |
| Future broad op built without the gate | uncontrolled mass mutation | system-wide |

Blast radius: **124G itself is zero** (audit/docs only — no runtime mutation). The
risk it *catalogues* is system-wide (the unbuilt executor), which is the accepted
residual.

## Round-1 gate findings (folded; triple-GREEN)

- **(all roles) Slice 4 evidence fields are target-state, not current inventory.**
  `materializerContractVersion`, `sourceFingerprint`, `schemaWidgetContractFingerprint`,
  `overlayFingerprint` exist nowhere in the codebase today (only `publicPackageFingerprint`
  is persisted); `ck-runtime-materializer` isn't built and 124B–124F are plan-green.
  So Slice 4's "confirm generated packages record … fingerprints required by 124A"
  is a forward claim against unbuilt code. Mark Slice 4's evidence table as
  target-state/deferred to stay honest with the no-reinterpretation tenet (audit
  current truth; don't assert future state as current). The 124A §9.1 / 124C
  carry-forward (compute/persist source/schema fingerprints) is what makes it real.
- **(Senior-PM) Operator staleness-detection surface is unsequenced.** The audit
  lets an operator reason about staleness *in principle*, but there's no
  "which of my N accounts are stale after this Dieter change" query — out of 124G
  scope (correctly routed to the future broad-re-resolution PRD), but the program
  should sequence that enumeration capability explicitly so it isn't an indefinite
  blind spot.
- **(Staff-eng/TPM) Sealed-vs-external classification verified code-accurate:**
  widget CSS/JS are inlined into package bytes (`account-instance-public-package.ts:271-292`);
  `/dieter/**` stays a live `@import` reference (:275). The Slice 2/3 split is a
  faithful model of today's bytes, not an invented abstraction.

## Recommendation
**Green-light.** 124G is the correct audit + gate slice: it classifies the broad-
dependency surface, defines sealed/external/evidence/unmarked rules, and gates any
future mass re-resolution behind a 10-field authority contract — while implementing
none of it (right discipline: contract before engine). Residual risk (no system-wide
executor yet; base-layer changes can mass-stale until a future PRD) is made explicit,
not hidden. One dependency: the evidence detection rests on 124C computing/persisting
source/schema fingerprints (the 124A §9.1 carry-forward). Forward to the three-role gate.
