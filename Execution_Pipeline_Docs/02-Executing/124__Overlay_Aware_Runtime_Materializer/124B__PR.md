## Verdict

**Green-light**, with four binding closeout conditions. 124B is the right shape
for a pure-package slice: it creates `packages/ck-runtime-materializer` as a pure
resolver, changes no runtime behavior, defers every 124A decision, and forbids
itself from being a Worker/service/agent/storage-writer/policy-engine. It honors
the no-reinterpretation tenet by *extracting* Roma's existing build logic rather
than inventing a renderer. Blast radius is low by design (the package is unwired
until 124C) — the real risk is contract-correctness for 124C/124D, plus one
action that touches a live system (the fingerprint hoist, below).

## Slice-by-slice review

### Slice 1 — Package scaffolding
Correct and boring. **Closeout condition:** the "no forbidden imports" rule (also
Slices 8/9) must be a CI-enforced structural check (eslint rule or an import-graph
test), not a manual review — the package's entire value is its isolation; a later
Roma/Tokyo/Supabase import would silently destroy purity.

### Slice 2 — Roma builder inventory
Sound. The extraction map is the right artifact. **Note:** Roma's builder is
tangled with I/O (compiled-widget fetch, Tokyo asset resolution) — the "pure
logic" boundary must be drawn explicitly; expect non-trivial seams around media
asset resolution and compiled-widget payload shaping. Capture the boundary in the
addendum, don't discover it in Slice 5.

### Slice 3 — Input contract
Sound. **Closeout condition (compiled-widget type boundary):** the materializer
must define its OWN compiled-widget input type in the package — it must NOT
import a type from `@clickeen/bob` (a service package). Roma (124C) adapts Bob's
compiled output to the materializer's input type. This keeps the materializer
Bob-free and makes the input contract owned by the package, not by Bob. List
`@clickeen/bob` explicitly in the forbidden-import set.

### Slice 4 — Output and evidence (THE crux slice)
- **Closeout condition (fingerprint hoist — load-bearing):** today **Tokyo**
  computes the package fingerprint (`buildInstancePublicPackageFingerprint`,
  `package-files.ts:93`); Roma computes none. 124B's materializer must emit a
  "deterministic generated package fingerprint" that **exactly matches** what
  Tokyo stores and the serve-gate checks (`clk-live-routes.ts:122,128`). The
  materializer cannot import from tokyo-worker (forbidden). Therefore the
  fingerprint function + `SubmittedInstancePublicPackage` type must be **hoisted
  to a shared package** (ck-contracts or ck-runtime-materializer itself), and
  tokyo-worker must import it from there.
  - This hoist is the **one 124B action that touches a live system** (Tokyo).
    It must be proven behavior-preserving: a golden test asserting identical
    fingerprint bytes for the same package before and after the hoist. Without
    it, 124C wiring would cause the serve-gate to 404 every package (materializer
    fingerprint ≠ Tokyo fingerprint).
  - Decide authority: does the materializer become the fingerprint producer
    (Tokyo trusts it) or does Tokyo keep recomputing on receipt (materializer's
    value is evidence/echo)? Either is fine, but pick one and state it. Either
    way: one shared function, used by both.
- **Two-hash rule (from 124A):** SHA-256 for package/artifact fingerprints;
  FNV-1a only for internal identity keys. Do not conflate.

### Slice 5 — Base locale resolver
Sound as a no-behavior-change extraction. **Closeout condition (golden parity):**
"preserve current behavior" **cannot be proven in 124B** — Roma does not call the
package yet (that is 124C). So 124B must prove parity by **capturing current
Roma base output as golden files and asserting byte-for-byte reproduction** in
package tests. Live runtime parity is 124C's gate, not 124B's. The PRD's Slice 8
test list should explicitly include a golden-output parity test against captured
current Roma bytes.

### Slice 6 — Locale overlay resolver
Correct. Correctly defers repeated-field support to 124A's `scalar_only_initially`
and explicitly non-claims unsupported repeated fields (V3 guard, no silent
omission). **Note:** the overlay-application semantics must match `previewL10n.js`'s
`applyTranslatedValues` exactly — that parity is owned by Slice 7.

### Slice 7 — Parity support
Sound. Inherits 124A's decision (shared resolver preferred; conformance-only if
runtime split forbids shared code). **Note:** per 124A Round-2, the conformance
suite must be CI-gated because preview (`previewL10n.js`, browser JS) and the
materializer (server TS) are two divergent implementations today.

### Slice 8 — Tests
Good coverage. **Add:** the golden-output parity test (Slice 5) and the
forbidden-imports structural test (Slice 1). **Resolve:** the FNV-1a
`widgetEditableFieldsContractHash` (wire vs remove) — don't ship dead code in a
contract package.

### Slice 9 — Closeout gate
Correct. Add: confirm the fingerprint hoist is byte-identical for Tokyo (golden
test) before 124B closes, since 124C cannot proceed if fingerprints disagree.

## Vectors and blast radius

## Recommendation

**Green-light.** 124B is a clean, low-blast-radius pure-package slice that honors
the no-reinterpretation tenet. Four binding closeout conditions, all within 124B's
scope: (1) **hoist `buildInstancePublicPackageFingerprint` to a shared package
and prove Tokyo byte-identical after hoist** — the one live-system risk and the
prerequisite for 124C; (2) materializer defines its own compiled-widget input
type, no `@clickeen/bob` import; (3) golden-output parity tests against captured
Roma bytes; (4) CI-enforced, precise forbidden-import rule (allow shared-contract
packages, forbid service/platform). Forward to the three-role gate.
