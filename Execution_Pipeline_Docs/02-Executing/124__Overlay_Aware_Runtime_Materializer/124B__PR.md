## Watch-items (non-blocking)

1. **Fingerprint byte-order consistency (cross-PRD).** 124B's
   `buildRuntimePackageFingerprint` uses "the deterministic byte order named by
   the 124A addendum." Tokyo's existing `buildInstancePublicPackageFingerprint`
   (`tokyo-worker/.../package-files.ts:93`) has its own byte order (length+content
   over the concatenated files). For 124C reconciliation to be possible *without
   changing Tokyo*, the 124A addendum's byte order should match Tokyo's current
   algorithm. 124B self-gates on 124A naming a byte order (good) — but the
   addendum should name *Tokyo's* order, not a new one. (Tolerable if they
   differ: 124B evidence is echo-only and Tokyo stays the serve authority per the
   124C gate — but matching avoids a future Tokyo change.)
2. **Single-fixture parity proves ONE widget shape.** Slice 5's fixture is one
   compiled widget. That proves the port is faithful for that shape's code paths
   (scalar, nested, social-share, conditional repeated). It does **not** prove
   parity across all widget types — `stampPackageRoot`/`resolveProductPath` are
   widget-shape-sensitive. Cross-widget-type parity is **124C's** gate (when Roma
   is wired in, dual-build across real cloud-dev configs). Acceptable for 124B's
   package-isolated scope; flagging so the single green fixture isn't mistaken
   for full runtime parity.
3. **`applyLocaleOverlayToState` consumer (Slice 2 export).** Exported because
   "124C/124D need the same resolver semantics." It's a TS package function; Bob
   preview runs browser JS (`previewL10n.js`). So this export serves the
   conformance suite / 124D / 124C — not browser preview directly — unless a
   later step bundles it for the browser. That's 124A's parity-mechanism call;
   124B just exports it. Note the consumer so it isn't assumed to be
   preview-usable as-is.
4. **`node:crypto` fallback (Slice 8).** Node 18+ exposes
   `globalThis.crypto.subtle`, so `crypto.subtle.digest` works under `tsx`; the
   `node:crypto` `createHash` fallback is likely dead. Harmless, but ensure the
   test covers whichever path actually runs (don't ship an untested branch).
