## 2026-06-04 System Tenets Correction

106G is allowed to be an edge-file delivery PRD.

106G is not allowed to make Tokyo the product publish authority.

Correct ownership:

- Roma owns publish/unpublish intent, account context, tier permission, and
  user-facing publish errors.
- Page Composer/Roma must produce or request the final page package before
  delivery state is made public.
- Tokyo stores delivery state and generated page files in R2.
- `clk.live` reads stored delivery state and serves allowlisted files.
- Tokyo must not compose the page, decide product eligibility, repair broken
  output, or synthesize SEO metadata at request time.

The public serving path remains valid:

```text
https://clk.live/{accountPublicId}/pages/{pageId}
```

The next 106G repair slice must ensure publish/unpublish code follows that
ownership. Serving already-stored files from R2 remains in Tokyo/clk.live.

## Purpose

Deliver materialized Clickeen page packages on the internet.

The physical files live in Clickeen storage:

```text
page index.html
page styles.css
page runtime.js
```

106G makes those files usable from:

- a Clickeen-hosted page URL;
- a one-line whole-page embed for Prague and customer sites;

This PRD does not introduce slugs, website routes, custom domains, redirects, or a site builder.

## Object-Addressed Public Coordinates

V1 uses object coordinates, not slugs.

Single widget:

```text
https://clk.live/{accountPublicId}/{instanceId}
```

Composed page:

```text
https://clk.live/{accountPublicId}/pages/{pageId}
```

The page id is the public Clickeen page coordinate.

Do not add:

- `pageSlug`;
- slug uniqueness;
- old-slug behavior;
- aliases;
- redirects;
- pretty route maps;
- custom domain routing.

Those belong to a later website-routing/custom-domain PRD.

## Publish Contract

106F writes the current page package:

```text
accounts/{accountPublicId}/website/publishes/{pageId}/
  index.html
  styles.css
  runtime.js
```

106G gates public delivery for that page id.

Publish flow:

```text
save page source
materialize page package through 106F
validate generated files
write page serve state = published
serve pageId from the current generated package
```

Unpublish flow:

```text
disable public delivery for pageId
clk.live/{accountPublicId}/pages/{pageId} returns 404
generated package files can remain for recomposition, preview, and future embed work
```

This is the same boring split as widgets:

```text
generated package exists = Clickeen can compose/preview/deliver after authorization
serve state published = public clk.live page may serve
```

The public route must not serve a page by raw object existence alone. A package can exist without being publicly delivered.

## Audit Refresh Decisions

The audit refresh is useful for 106G because it confirms the current `clk.live` route only understands widget coordinates:

```text
/{accountPublicId}/{instanceId}/{file?}
```

106G must add the page coordinate explicitly:

```text
/{accountPublicId}/pages/{pageId}/{file?}
```

Use:

- page delivery must read page serve state before any generated page file;
- raw R2 package existence must never make a page public;
- existing single-widget delivery must not regress;
- public serving must keep the same route-safety discipline already present in `clk-live-routes.ts`.

Keep the current V1 package path decision:

```text
accounts/{accountPublicId}/website/publishes/{pageId}/
```

Do not add `publishId`, slug route maps, aliases, or "latest publish" lookup. Source and page serve state live under `website/pages/{pageId}/`; the generated current page package lives under `website/publishes/{pageId}/`.

Implementation may factor page parsing into a helper or separate route file, but it must remain one public delivery service, not a second public delivery subsystem.

## Delivery Modes

### 1. Clickeen-Hosted Page

The hosted page URL serves the whole composed page:

```text
https://clk.live/{accountPublicId}/pages/{pageId}
```

Support files are served from the same object coordinate:

```text
https://clk.live/{accountPublicId}/pages/{pageId}/styles.css
https://clk.live/{accountPublicId}/pages/{pageId}/runtime.js
```

This is useful for:

- Clickeen-hosted landing pages;
- previews;
- customer sharing;
- dogfooding from Prague;
- future custom domains that can later resolve to the same page id.

### 2. Whole-Page Embed

Customers and Prague must be able to inject a composed Clickeen page with one line.

Candidate V1 shape:

```html
<script src="https://clk.live/{accountPublicId}/pages/{pageId}/embed.js"></script>
```

The embed loads the active page package and renders the whole composed page into the host document.

This supports:

- Prague using the same system Clickeen sells;
- WordPress pages/posts;
- Shopify pages/sections where custom script/embed is allowed;
- Squarespace/custom HTML blocks;
- any customer site that accepts a script embed.

Do not implement this as one iframe per widget.

If an iframe is used as a fallback, it is one iframe for the whole composed page, not stacked widget iframes.

Whole-page embed is useful, but it is not allowed to expand 106G into CMS/plugin support.

V1 embed scope:

- serve one delivery script from the active page package coordinate;
- inject the already-materialized whole page package;
- no WordPress/Shopify/Squarespace plugin logic;
- no customer-domain SEO claim;
- no placement-level implementation.

### 3. Placement Embed Is Not V1

106G does not implement selected-placement delivery.

106F page packages preserve stacking order with plain structural wrappers, not placement ids.

If product later needs selected-placement delivery, that work must first define a deliberate selection contract. It must not retrofit hidden placement IDs into 106E/106F/106G and must not invent blocks or sections.

The V1 public surface is the whole hosted page and the whole-page embed.

## Edge Serving Rules

Edge route serves only generated page delivery assets:

```text
index.html
styles.css
runtime.js
embed.js
```

It must not serve:

- page source;
- page indexes;
- page serve-state files;
- website manifest/management files;
- locale overlay files;
- instance source files;
- account asset metadata;
- internal API output.

Required behavior:

- `GET` and `HEAD` for hosted page/support files where applicable;
- HTTPS redirect where applicable;
- traversal/encoded-slash/backslash rejection;
- correct content type;
- `x-content-type-options: nosniff`;
- deliberate cache headers;
- `404` for unpublished, missing, disabled, or inaccessible pages;
- existing single-widget URL behavior does not regress.

## SEO/GEO Baseline

For the Clickeen-hosted page URL, `index.html` must include:

- primary visible content in initial HTML;
- title;
- meta description;
- canonical URL based on `https://clk.live/{accountPublicId}/pages/{pageId}`;
- robots directive;
- normal crawlable links where links exist;
- valid structured data only when it matches visible content.

Do not generate SEO copy at request time.

Do not inject metadata only after JavaScript runs.

Do not add crawler-only hidden content.

### Customer-Site Embed SEO Reality

A client-side script embed can visually place Clickeen content into WordPress, Shopify, Squarespace, or Prague.

That is not the same as guaranteeing first-paint SEO content on the customer domain.

Customer-domain SEO/GEO requires one of:

- host platform/server-side rendering through a plugin;
- a custom-domain/proxy delivery model;
- another explicitly scoped integration where the customer page HTML contains the Clickeen content before JavaScript.

Do not overclaim SEO/GEO for generic script embeds.

The hosted `clk.live` page must be technically crawlable. Customer-domain SEO delivery is a later integration/custom-domain decision unless explicitly added to V1.
