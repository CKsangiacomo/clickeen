# PRD 082 ‚Äî System Health, Taxonomy, And Stress Audit

Status: Planning
Created: 2026-05-02
Repo commit audited: `ba575c30`

## Intent

Run a whole-system sanity pass before more feature work.

This is not a feature PRD. It is a cleanup and proof plan:

1. Make the codebase taxonomy match the actual product.
2. Delete tracked garbage, orphan files, dead local folders, and fake product concepts.
3. Stress the real runtime shape before trusting it at scale.

## Product Truth

The surviving product path is:

`account opens widget in Roma -> Bob edits one widget in one active locale -> Roma saves to Tokyo -> async systems handle follow-up work`

The system must not preserve fake product modes just because code exists for them.

Surviving authorities:

| Concern | Authority |
|---|---|
| Auth/session/login identity | Berlin + Supabase RPC |
| Account membership/policy | Berlin + `ck-policy` account policy |
| Widget authoring host | Roma |
| Widget editor | Bob |
| Account assets and saved config objects | Tokyo-worker + R2 |
| Published public embed reads | Venice |
| AI execution | San Francisco |
| Marketing/demo/funnel pages | Prague |
| Database truth | Supabase/Michael |
| San Francisco internal telemetry | D1 |

## Non-Goals

- Do not delete Supabase migrations. They are append-only history.
- Do not delete R2 objects without a DB-backed inventory diff.
- Do not rewrite working services just to make files prettier.
- Do not create broad abstractions before deleting known residue.
- Do not preserve old route names, folders, or types if they only support a dead product model.

## Concrete Findings

### Finding 1 ‚Äî Tracked Ignored Virtualenv And Inbox

`git ls-files -ci --exclude-standard` returns `5217` tracked ignored files.

Almost all are under:

- `tooling/whisper/.venv/**`
- `tooling/whisper/inbox/**`

`tooling/whisper` is about `321M` on disk.

This is not product code. It is a major repo and AI-context pollutant.

- Keep only lightweight source files if the tool is still useful.

- CI: none expected unless something incorrectly depends on committed venv files.

## Storage Taxonomy Target

Tokyo static repo shape should stay simple:

```txt
tokyo/
  product/
    assets/
    dieter/
    fonts/
    themes/
    widgets/
  roma/
    i18n/
  prague/
    assets/
    l10n/
    pages/
  accounts/
    README.md
```

R2 shape should be account-first for user-owned state:

```txt
accounts/{account_id}/assets/{asset_id}/{filename}
accounts/{account_id}/widgets/{public_id}/draft/config.json
accounts/{account_id}/widgets/{public_id}/published/{revision}/...
accounts/{account_id}/l10n/{public_id}/{locale}/...
```

Rules:

- No workspace prefixes.
- No generic global `assets/` bucket prefix for account-owned files.
- No templates prefix as a product truth.
- No per-user folder below account unless the product actually owns per-user files.
- No R2 cleanup without comparing keys against Supabase rows.

## D1 Versus Supabase Rule

Supabase/Michael owns relational product truth:

- accounts
- users
- memberships
- widget registry rows
- asset registry rows
- usage events that drive product policy
- auth identity resolution

D1 is acceptable only for service-local operational state:

- San Francisco AI telemetry
- San Francisco request/outcome logs
- bounded queues or internal worker state when the service owns the lifecycle

D1 must not become a second product database.

## Stress-Test Plan

### Static Health Gates

Run before product stress:

```sh
pnpm lint
pnpm typecheck
pnpm test
pnpm build
git diff --check
git ls-files -ci --exclude-standard
```

Target:

- No tracked ignored files.
- No stale workspace naming in active product policy.
- No active legacy route aliases unless explicitly documented.

### Product Smoke Tests

Run against deployed dev after each cleanup batch:

1. Roma login creates/resolves one account identity.
2. Roma bootstrap loads account/policy once.
3. Widgets page lists account instances and listed starter instances.
4. Builder opens an account-owned instance.
5. Bob edits one active locale.
6. Roma saves to Tokyo.
7. Asset upload succeeds, usage updates, quota errors are specific.
8. Uploaded asset appears in Builder picker and renders in preview.
9. Translation generation starts after save and resolves to available locales.
10. Published widget is served by Venice.
11. Unpublished widget is unavailable from public embed.

### Scale Tests

These tests prove the architecture can survive real use.

Account scale:

- Seed or simulate 1M accounts.
- Average 2-3 widgets per account.
- Average 5 instances for the FAQ widget.
- Verify account-scoped queries use indexes and do not scan global tables.

Asset scale:

- Simulate accounts with many assets.
- Verify Tokyo-worker lists by account prefix only.
- Verify no code path globally lists the R2 bucket for a request.
- Verify quota calculation is driven by indexed account asset rows or bounded account-prefix inventory, not global storage scans.

Translation scale:

- Save a widget with changed text.
- Queue translation for multiple locales.
- Verify idempotency by repeating the same save.
- Verify retries stop at a bounded state.
- Verify UI never stays in "preparing" when work has failed or completed.
- Verify existing translated paths are preserved when only changed paths are regenerated.

Auth scale:

- Send concurrent first-login requests for the same identity.
- Verify the DB RPC returns one winning user/account identity.
- Verify no duplicate account or membership rows.

Public embed scale:

- Hammer Venice public reads for published widgets.
- Verify no Supabase/Paris calls on hot public reads.
- Verify locale miss returns unavailable, not fallback magic.

AI cost scale:

- Simulate AI copilot and translation usage.
- Verify limits are enforced at the point cost is incurred.
- Verify usage writes are product-readable for SaaS quota UI.
